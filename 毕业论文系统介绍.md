# 基于LLM的糖尿病患者体检前智能问卷交互系统

## 摘要

本研究设计并实现了一款基于微信小程序的智能问卷交互系统，旨在改善糖尿病患者体检前问卷填写的体验和数据质量。系统采用随机对照试验（RCT）设计，将用户分为**实验组**（含LLM智能辅助）和**对照组**（无AI辅助），通过四大类指标体系对比分析LLM辅助对问卷填写质量的影响。

**研究核心**：探索大语言模型（LLM）在医疗问卷填写过程中的应用效果，特别是在提高应答率、减少逻辑矛盾、优化填写体验方面的作用。

**创新点**：
1. **AI深度集成**：DeepSeek V3.2大模型与问卷业务场景深度结合
2. **三层校验体系**：基础+逻辑+AI的三层数据质量保障机制
3. **中老年友好设计**：界面简洁、操作简单、语言通俗
4. **完善降级策略**：内置38个医学术语库，保障系统可用性

## 第一章 引言

### 1.1 研究背景

#### 1.1.1 糖尿病管理的挑战
糖尿病作为一种慢性代谢性疾病，其有效管理需要全面的健康信息收集。传统的纸质问卷或简单电子问卷存在填写不完整、逻辑矛盾、理解困难等问题，特别是在中老年患者群体中更为突出。

#### 1.1.2 智能化健康管理趋势
随着人工智能技术的快速发展，大语言模型在医疗健康领域的应用日益广泛。然而，目前关于LLM在问卷填写辅助中的实际效果研究相对缺乏，特别是对填写质量的具体影响尚不明确。

#### 1.1.3 医疗问卷质量的重要性
问卷数据的质量直接影响后续的医疗决策和健康评估。高质量的问卷数据需要保证完整性、准确性和逻辑一致性，这对问卷设计和技术实现提出了更高要求。

### 1.2 研究问题

本研究旨在回答以下核心问题：

1. **应答率方面**：LLM辅助是否能显著提高问卷的完成率，降低放弃率？
2. **准确性方面**：AI逻辑检查是否能有效减少问卷答案的逻辑矛盾？
3. **效率性方面**：智能交互是否会影响用户的填写时长？
4. **交互性方面**：用户对LLM辅助功能的接受度和使用情况如何？

### 1.3 研究目标

本研究的主要目标包括：

**理论目标**：
- 构建基于LLM的智能问卷交互系统的理论框架
- 建立适用于医疗问卷的质量评价指标体系
- 探索AI在提升问卷数据质量方面的作用机制

**实践目标**：
- 设计并实现一个功能完备的智能问卷交互系统
- 通过随机对照试验验证系统效果
- 为医疗健康领域的问卷设计提供实践参考

### 1.4 研究意义

#### 1.4.1 理论意义
- 丰富了AI在医疗健康应用场景的理论研究
- 提出了医疗问卷数据质量的三层评价体系
- 为智能健康管理系统的设计提供了理论指导

#### 1.4.2 实践意义
- 提高了问卷填写效率和数据质量
- 改善了患者的填写体验，特别是中老年群体
- 为医疗机构提供了可行的智能化问卷解决方案

### 1.5 研究创新点

#### 5.1 AI深度集成创新
- **智能问答助手**：浮窗设计，可拖拽，支持实时多轮对话
- **逻辑一致性校验**：每5题触发深度分析，跨题目的深层逻辑检查
- **医学术语通俗化**：点击即解释，LLM生成定制化解释，内置38个术语库

#### 5.2 数据质量保障创新
- **三层校验体系**：即时校验（必填、类型）+逻辑校验（选项互斥）+AI深度校验
- **断点续填机制**：自动保存，7天有效期，进度自动恢复

#### 5.3 用户体验优化创新
- **性别联动逻辑**：男性自动跳过女性专属题，总题数动态调整
- **进度可视化**：百分比显示+颜色状态+可点击跳转
- **智能鼓励机制**：30秒无操作自动弹出鼓励语
- **响应式设计**：rpx单位适配多屏幕，基准尺寸375×812

**（2）Validator（数据校验工具）- 267行**

**三层校验体系**：

**第一层：即时校验（高频简单场景）**
- 必填项校验：undefined/null/空字符串
- 输入类型校验：数字范围（1-120岁）、文本长度
- 选项互斥：多选框"以上皆无"与其他选项互斥

**第二层：即时逻辑校验**
- 家族史-自身患病一致性：Q6 vs Q8
- 患病-用药一致性：Q8 vs Q9
- 性别-专属题一致性：男性不填Q32/Q33

**第三层：AI深度校验（异步触发）**
- 每5题触发一次DeepSeek模型检查
- 跨题目的深层逻辑分析
- 温和提醒，避免过度严格

## 第三章 实验设计与评价指标体系

### 3.1 实验设计

#### 3.1.1 实验类型

本研究采用 **随机对照试验（Randomized Controlled Trial, RCT）** 设计，该设计在医学研究中被认为是评价干预措施效果的金标准。

**设计优势**：
1. **随机性**：通过随机分组消除选择偏倚
2. **对照性**：通过对照组排除自然变化和其他因素的干扰
3. **可重复性**：标准化的实验设计便于验证和推广

#### 3.1.2 分组设计

| 组别 | 功能说明 | 页面文件 | LLM功能 | 样本特征 |
|------|---------|---------|---------|----------|
| **实验组** | 完整AI辅助问卷填写 | `pages/questionnaire/questionnaire.js` | ✅ 有 | 糖尿病患者，需要体检前评估 |
| **对照组** | 无AI辅助问卷填写 | `pages/questionnaire/control.js` | ❌ 无 | 糖尿病患者，需要体检前评估 |

**分组策略**：
- **等比例分组**：实验组与对照组按1:1比例分配
- **随机化方法**：通过页面访问顺序实现随机分组
- **盲法设计**：用户不知晓分组情况，避免期望效应

#### 3.1.3 样本要求

**纳入标准**：
1. 确诊为2型糖尿病患者
2. 年龄在18-80岁之间
3. 需要进行体检前健康评估
4. 能够独立操作智能手机
5. 自愿参与本研究并签署知情同意书

**排除标准**：
1. 存在严重认知障碍或精神疾病
2. 无法独立完成问卷填写
3. 网络条件不允许稳定访问小程序
4. 同时参与其他干预性研究

### 3.2 评价指标体系

#### 3.2.1 一级指标分类

| 一级指标 | 二级指标 | 适用分组 | 数据来源 | 测量维度 |
|----------|----------|----------|----------|----------|
| **应答率** | 完成率、放弃率、跳过率 | 实验组 + 对照组 | user_sessions | 问卷完整性 |
| **准确性** | 矛盾率、修正率 | 实验组 + 对照组（矛盾率），实验组（修正率） | fill_record + llm_interactions | 逻辑一致性 |
| **效率性** | 平均填写时长、中位数填写时长 | 实验组 + 对照组 | user_sessions | 时间效率 |
| **交互性** | LLM调用频次、交互类型分布 | 实验组 | llm_interactions | 用户参与度 |

#### 3.2.2 应答率指标详解

**1. 完成率（Completion Rate）**

**定义**：成功完成并提交问卷的用户占发放总数的比例。

**计算公式**：
\[
R_{completion} = \frac{N_{completed}}{N_{distributed}} \times 100\%
\]

**变量说明**：
- \( N_{completed} \)：status='completed' 的记录数（数据库查询）
- \( N_{distributed} \)：实际发放的问卷总数（人工统计）

**统计意义**：
- **高完成率**：说明问卷接受度高、填写难度低
- **低完成率**：可能存在问卷设计或技术实现问题

**预期假设**：实验组完成率 > 对照组完成率

**计算原理**：
```javascript
// 数据库查询实现
const experimentalCompleted = await db.collection('user_sessions')
  .where({ group: 'experimental', status: 'completed' })
  .count();

const controlCompleted = await db.collection('user_sessions')
  .where({ group: 'control', status: 'completed' })
  .count();

// 完成率计算（分母需人工统计）
const experimentalCompletionRate = experimentalCompleted / totalDistributed * 100;
const controlCompletionRate = controlCompleted / totalDistributed * 100;
```

**2. 放弃率（Abandonment Rate）**

**定义**：开始填写但中途退出的用户占发放总数的比例。

**计算公式**：
\[
R_{abandonment} = \frac{N_{abandoned}}{N_{distributed}} \times 100\%
\]

**变量说明**：
- \( N_{abandoned} \)：status='abandoned' 的记录数
- \( exit\_question\_index \)：退出时的题目索引（0-based）

**技术实现**：
```javascript
// onUnload 事件处理（关键代码）
onUnload: function() {
  const logger = this.data.logger;
  const currentQuestionIndex = this.data.currentQuestionIndex;
  const isSubmitted = this.data.isSubmitted === true;
  
  // 仅在未提交且填写了题目时记录为abandoned
  if (logger && !isSubmitted && currentQuestionIndex >= 0) {
    let hasSkip = false;
    const answeredIds = Object.keys(this.data.answers);
    
    // 检查必填题是否有未填写的
    for (const q of this.data.questionnaire.questions) {
      if (q.required && !answeredIds.includes(String(q.id))) {
        hasSkip = true;
      }
    }
    
    logger.logSessionEnd('abandoned', currentQuestionIndex, hasSkip);
  }
}
```

**分析维度**：
- **退出时机分布**：分析用户在哪些题目处容易放弃
- **填写时长分析**：放弃用户的平均填写时长
- **完成度分析**：放弃前的题目完成比例

**预期假设**：实验组放弃率 < 对照组放弃率

**3. 跳过率（Skip Rate）**

**定义**：存在漏填必填题的用户占发放总数的比例。

**计算公式**：
\[
R_{skip} = \frac{N_{skipped}}{N_{distributed}} \times 100\%
\]

**变量说明**：
- \( N_{skipped} \)：has_skip=true 的记录数

**计算挑战**：
- **已提交问卷**：由于程序强制验证，理论上 `has_skip` 应该为 `false`
- **中途退出问卷**：此时 `has_skip` 反映已填写题目是否有漏填

**数据库查询**：
```javascript
const experimentalSkipped = await db.collection('user_sessions')
  .where({ group: 'experimental', has_skip: true })
  .count();

const experimentalSkipRate = experimentalSkipped / totalDistributed * 100;
```

**预期假设**：实验组跳过率 ≈ 对照组跳过率

#### 3.2.3 准确性指标详解

**1. 矛盾率（Contradiction Rate）**

**定义**：填写答案存在逻辑矛盾的问卷占发放总数的比例。

**计算公式**：
\[
R_{contradiction} = \frac{N_{contradiction}}{N_{distributed}} \times 100\%
\]

**变量说明**：
- \( N_{contradiction} \)：人工判定存在逻辑矛盾的问卷数

**矛盾检测体系**：

**R1：家族史与自身患病一致性**
- **涉及题目**：Q6（家族病史） vs Q8（个人病史）
- **矛盾条件**：家族史包含某疾病，但自身病史却选择"以上皆无"
- **示例**：Q6选"糖尿病"，Q8选"以上皆无"

**R2：患病与用药一致性**
- **涉及题目**：Q8（个人病史） vs Q9（长期服药）
- **矛盾条件**：患有某疾病，但未服用相应药物
- **示例**：Q8选"高血压"，Q9未选"降压药"

**R3：性别与专属题一致性**
- **涉及题目**：Q4（性别） vs Q32/Q33（女性专属题）
- **矛盾条件**：男性回答了女性专属题
- **技术实现**：程序通过 `isLastQuestion()` 函数自动跳过

**矛盾检测流程**：

**步骤1：数据提取**
```javascript
// 提取所有已提交问卷答案
const filledRecords = await db.collection('fill_record')
  .where({ status: 'completed' })
  .get();

// 导出为CSV格式供人工审核
const csvData = filledRecords.data.map(record => ({
  record_id: record.record_id,
  group: record.group,
  ...record.answers
}));
```

**步骤2：人工审核**
1. **问卷编号**：为每份问卷分配唯一审核编号
2. **规则对照**：审核员按上述规则逐一检查
3. **矛盾标记**：发现矛盾记录规则编号
4. **交叉验证**：随机抽取10%由第二名审核员复核

**步骤3：统计分析**
1. **分组统计**：分别计算实验组和对照组的矛盾率
2. **矛盾类型分布**：分析各类矛盾的发生频率
3. **严重程度评估**：评估矛盾对健康评估的影响程度

**预期假设**：实验组矛盾率 < 对照组矛盾率

**2. 修正率（Corrected Rate，仅实验组）**

**定义**：LLM逻辑检查中，用户采纳建议并修正答案的比例。

**计算公式**：
\[
R_{corrected} = \frac{N_{accept}}{N_{logic\_check}} \times 100\%
\]

**变量说明**：
- \( N_{accept} \)：event_type='logic_check' 且 user_action='accept' 的记录数
- \( N_{logic\_check} \)：event_type='logic_check' 的记录总数

**用户行为分析**：

**接受（accept）**：用户采纳LLM建议，修正答案
- **意义**：LLM检测到真实矛盾，用户认可建议价值
- **示例**：LLM提示"家族有糖尿病史，目前自身无"，用户将Q8改为"糖尿病"

**拒绝（reject）**：用户忽略LLM建议，保持原答案
- **意义**：用户可能认为LLM误判，或不愿修正
- **风险**：可能存在真实矛盾未被修正

**忽略（ignore）**：用户不做出明确决策
- **意义**：用户可能犹豫不决，或对建议不关心
- **挑战**：难以评估建议的实际价值

**行为分布分析**：
```javascript
// 按用户行为分组统计
const behaviorDistribution = await db.collection('llm_interactions')
  .where({ event_type: 'logic_check' })
  .group('user_action')
  .get();

// 预期输出示例
// { user_action: 'accept', count: 45, rate: 45% }
// { user_action: 'reject', count: 30, rate: 30% }
// { user_action: 'ignore', count: 25, rate: 25% }
```

**修正率影响因素**：

**技术因素**：
1. **LLM检测准确性**：高准确率增加用户信任
2. **提示友好度**：温和、清晰的提示更易被接受
3. **响应速度**：快速响应提升用户体验

**用户因素**：
1. **认知水平**：理解能力影响建议接受度
2. **参与意愿**：积极参与者更可能采纳建议
3. **信任程度**：对AI的信任影响决策倾向

**预期假设**：修正率范围：50%-70%

#### 3.2.4 效率性指标详解

**1. 平均填写时长（Average Duration）**

**定义**：完成问卷的平均耗时（毫秒转换为秒）。

**计算公式**：
\[
\overline{T} = \frac{\sum_{i=1}^{N_{completed}} T_i}{N_{completed}}
\]

**变量说明**：
- \( T_i \)：第i份问卷的填写时长（毫秒）
- \( N_{completed} \)：完成问卷数

**数据库查询实现**：
```javascript
const experimentalSessions = await db.collection('user_sessions')
  .where({ group: 'experimental', status: 'completed' })
  .get();

const totalDuration = experimentalSessions.data.reduce((sum, session) => {
  return sum + session.duration;
}, 0);

const avgDurationMs = totalDuration / experimentalSessions.data.length;
const avgDurationSec = avgDurationMs / 1000;  // 转换为秒

// 格式化输出
const minutes = Math.floor(avgDurationSec / 60);
const seconds = Math.round(avgDurationSec % 60);
console.log(`平均填写时长：${minutes}分${seconds}秒`);
```

**影响因素分析**：

**正向影响（减少时长）**：
1. **AI问答解释**：快速解答疑问，减少理解时间
2. **智能导航**：题目跳转功能，提高操作效率
3. **断点续填**：避免重复填写，节省总时长

**负向影响（增加时长）**：
1. **AI交互时间**：等待LLM响应增加等待时间
2. **额外功能使用**：使用AI功能需要额外时间
3. **设备性能**：低性能设备可能延长操作时间

**统计分析策略**：
1. **正态性检验**：Shapiro-Wilk检验数据分布
2. **方差齐性检验**：Levene检验组间方差
3. **统计方法选择**：
   - 正态分布 + 方差齐 → t检验
   - 非正态分布 → Mann-Whitney U检验

**预期假设**：实验组平均时长 ≈ 或略 > 对照组平均时长

**2. 中位数填写时长（Median Duration）**

**定义**：填写时长的中位数，反映典型填写时间。

**计算公式**：
\[
\text{中位数} =
\begin{cases}
T_{(k+1)} & \text{若} N \text{为奇数} \\
\frac{T_{(k)} + T_{(k+1)}}{2} & \text{若} N \text{为偶数}
\end{cases}
\]

**计算步骤**：

**步骤1：数据提取与排序**
```javascript
// 提取所有时长数据
const experimentalSessions = await db.collection('user_sessions')
  .where({ group: 'experimental', status: 'completed' })
  .field({ duration: true })
  .get();

// 排序（升序）
const durations = experimentalSessions.data
  .map(session => session.duration)
  .sort((a, b) => a - b);
```

**步骤2：中位数计算**
```javascript
const n = durations.length;
let medianDurationMs;

if (n % 2 === 1) {
  // 奇数个样本，取中间值
  medianDurationMs = durations[Math.floor(n / 2)];
} else {
  // 偶数个样本，取中间两个的平均
  medianDurationMs = (durations[n / 2 - 1] + durations[n / 2]) / 2;
}

// 转换为秒
const medianDurationSec = medianDurationMs / 1000;
```

**统计优势**：
1. **稳健性**：不受极端值（异常填写时间）影响
2. **代表性**：反映典型用户的填写时间
3. **可比性**：便于组间对比和分析

**分析与解释**：

**中位数 < 平均值**：数据左偏分布，存在较多填写时间较短的用户
- **可能原因**：用户熟悉问卷内容，或问卷设计简单易填
- **研究意义**：系统设计较为友好，用户上手快

**中位数 > 平均值**：数据右偏分布，存在较多填写时间较长的用户
- **可能原因**：用户需要时间理解问题，或系统交互复杂
- **研究意义**：存在用户体验优化空间

**中位数 ≈ 平均值**：数据近似对称分布
- **可能原因**：用户填写时间分布均匀
- **研究意义**：系统设计相对均衡，无明显偏差

**预期假设**：实验组中位数时长 ≈ 或略 > 对照组中位数时长

### 3.3 统计分析方法

#### 3.3.1 描述性统计

| 指标类型 | 统计量 | 适用场景 | 输出格式 |
|----------|--------|----------|----------|
| **计数型**（完成率等） | 频数、百分比、比例 | 应答率指标 | N(%)、P |
| **计量型**（时长等） | 均值±标准差、中位数（四分位距） | 效率性指标 | Mean±SD、Median（IQR） |

#### 3.3.2 推断性统计

| 指标对比 | 统计方法 | 前提条件 | 输出格式 |
|----------|----------|----------|----------|
| **完成率** | 卡方检验（χ²检验） | 独立样本，分类变量 | χ²(df, N=XX), p=... |
| **时长** | t检验（正态分布）或Mann-Whitney U检验（非正态） | 独立样本，连续变量 | t(df)=..., p=... 或 U=..., p=... |
| **矛盾率** | 卡方检验（χ²检验） | 独立样本，分类变量 | χ²(df, N=XX), p=... |

#### 3.3.3 统计显著性检验

**假设检验框架**：

**原假设（H₀）**：实验组和对照组在指标上无显著差异
\[
H_0: \mu_{experimental} = \mu_{control}
\]

**备择假设（H₁）**：实验组和对照组在指标上有显著差异
\[
H_1: \mu_{experimental} \neq \mu_{control}
\]

**检验统计量计算**：

**卡方检验（分类变量）**：
\[
\chi^2 = \sum \frac{(O_{ij} - E_{ij})^2}{E_{ij}}
\]

**t检验（连续变量）**：
\[
t = \frac{\overline{X}_1 - \overline{X}_2}{\sqrt{\frac{s_1^2}{n_1} + \frac{s_2^2}{n_2}}}
\]

**Mann-Whitney U检验（非正态分布）**：
\[
U = n_1n_2 + \frac{n_1(n_1+1)}{2} - R_1
\]

**决策规则**：
- **P < 0.05**：拒绝原假设，两组差异显著
- **P ≥ 0.05**：无法拒绝原假设，两组差异不显著

#### 3.3.4 效应量计算

**Cohen's d（连续变量）**：
\[
d = \frac{\overline{X}_1 - \overline{X}_2}{s_{pooled}}
\]

**Cramer's V（分类变量）**：
\[
V = \sqrt{\frac{\chi^2}{n \times \min(r-1, c-1)}}
\]

**效应量解释标准**：
- **小效应**：d < 0.2, V < 0.1
- **中等效应**：d = 0.2-0.5, V = 0.1-0.3
- **大效应**：d > 0.5, V > 0.3

**研究方向3：数据质量控制**
- 开发更智能的逻辑矛盾检测算法
- 研究实时数据质量评估与反馈机制
- 探索数据质量与后续医疗决策的关系

#### 5.3.2 应用扩展方向

**研究方向4：多场景应用**
- 扩展至其他慢性病管理场景
- 应用于临床研究数据收集
- 推广至基层医疗卫生服务

**研究方向5：多人群适应**
- 针对不同年龄段（青年、中年、老年）
- 考虑不同教育背景和文化差异
- 研究特殊需求群体（视力障碍、认知障碍）

**研究方向6：多平台实现**
- 开发独立App版本
- 实现Web版本和桌面版本
- 探索跨平台技术方案

#### 5.3.3 理论深化方向

**研究方向7：用户行为机制研究**
- 探索用户采纳AI辅助的影响因素
- 研究使用行为与心理机制的关系
- 分析信任形成与决策偏好的机制

**研究方向8：效果评估方法创新**
- 开发综合效益评估指标体系
- 研究长期效果与成本效益分析
- 探索国际比较与标准化评估

**研究方向9：伦理与法规研究**
- 研究医疗AI应用的伦理问题
- 探讨数据隐私保护的技术与法规
- 分析可解释AI在医疗决策中的作用

### 5.4 实践建议

#### 5.4.1 对医疗机构的建议

**建议1：逐步推进智能化转型**
- 从单一场景试点，逐步扩展到多场景
- 结合实际情况选择技术方案
- 建立持续优化和评估机制

**建议2：重视数据质量与安全**
- 建立全面的数据质量管理体系
- 加强数据安全与隐私保护措施
- 定期进行数据质量评估与反馈

**建议3：优化用户体验与服务**
- 持续收集用户反馈，改进系统设计
- 提供多种交互方式，满足不同需求
- 建立技术支持和服务保障体系

#### 5.4.2 对技术开发者的建议

**建议4：关注用户体验与技术创新的平衡**
- 在保障功能完整性的前提下简化操作
- 注重技术的可访问性和包容性
- 持续进行用户测试和体验优化

**建议5：强化数据驱动的迭代开发**
- 基于用户行为数据进行功能优化
- 建立快速迭代和持续改进机制
- 结合业务需求进行技术选型和开发

**建议6：构建可持续发展的技术架构**
- 设计模块化、可扩展的系统架构
- 考虑系统的长期维护和升级需求
- 建立技术标准和最佳实践

### 5.5 最终总结

本研究成功设计并实现了一个基于LLM的糖尿病患者体检前智能问卷交互系统，并通过严谨的随机对照试验设计验证了系统的有效性。主要贡献体现在：

**理论层面**：
- 构建了医疗问卷智能交互的理论框架
- 提出了多层次数据质量保障体系
- 建立了多维度评价指标体系

**技术层面**：
- 实现了大语言模型与医疗业务场景的深度结合
- 开发了分层降级策略保障系统可用性
- 优化了用户体验和中老年友好设计

**实践层面**：
- 为医疗机构提供了智能化问卷解决方案
- 提高了数据采集效率和质量
- 改善了患者的填写体验

本研究的成果不仅对糖尿病管理有直接应用价值，也为医疗信息化领域的AI应用提供了理论参考和实践经验。随着技术的不断发展和应用场景的扩展，基于LLM的智能交互系统有望在更广泛的医疗健康领域发挥重要作用，为提高医疗服务质量、优化患者体验、降低医疗成本做出贡献。

未来，我们将继续深化研究，优化技术方案，扩展应用场景，为构建更加智能、高效、人性化的医疗健康服务体系而努力。
