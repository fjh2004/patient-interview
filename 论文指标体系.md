# 基于LLM的智能问卷交互系统评价指标体系

## 一、研究概述

本研究采用随机对照试验（RCT）设计，将参与者随机分为**实验组**（使用LLM辅助问卷填写）和**对照组**（无AI辅助），通过量化指标对比分析LLM智能交互系统对问卷填写质量、效率和用户行为的影响。

---

## 二、指标体系框架

### 2.1 一级指标分类

| 一级指标 | 二级指标 | 适用分组 | 数据来源 |
|----------|----------|----------|----------|
| **应答率** | 完成率、放弃率、跳过率 | 实验组 + 对照组 | user_sessions |
| **准确性** | 矛盾率、修正率 | 实验组 + 对照组（矛盾率），实验组（修正率） | fill_record + llm_interactions |
| **效率性** | 平均填写时长、中位数填写时长 | 实验组 + 对照组 | user_sessions |
| **交互性** | LLM调用频次、交互类型分布 | 实验组 | llm_interactions |

---

## 三、应答率指标

### 3.1 完成率（Completion Rate）

**定义**：成功完成并提交问卷的用户占发放总数的比例。

**计算公式**：

\[
\text{完成率} = \frac{\text{提交问卷的用户数}}{\text{问卷发放总数}} \times 100\%
\]

**符号表示**：

\[
R_{completion} = \frac{N_{completed}}{N_{distributed}} \times 100\%
\]

其中：
- \( N_{completed} \)：status='completed' 的记录数
- \( N_{distributed} \)：实际发放的问卷总数（人工统计）

**计算方法原理**：

1. **统计分母**：通过问卷发放渠道（如医院现场分发、微信群推送等）统计实际发放的总问卷数
2. **统计分子**：从数据库 `user_sessions` 集合中查询 `status='completed'` 的记录数
3. **计算比例**：分子除以分母得到完成率

**数据库查询方法**：

```javascript
// 实验组完成数
const experimentalCompleted = await db.collection('user_sessions')
  .where({
    group: 'experimental',
    status: 'completed'
  })
  .count();

// 对照组完成数
const controlCompleted = await db.collection('user_sessions')
  .where({
    group: 'control',
    status: 'completed'
  })
  .count();

// 完成率计算
const experimentalCompletionRate = experimentalCompleted / totalDistributed * 100;
const controlCompletionRate = controlCompleted / totalDistributed * 100;
```

**统计意义**：
- 完成率反映问卷的可接受度和用户完成意愿
- 实验组预期完成率高于对照组，假设LLM辅助能降低填写难度

---

### 3.2 放弃率（Abandonment Rate）

**定义**：开始填写但中途退出的用户占发放总数的比例。

**计算公式**：

\[
\text{放弃率} = \frac{\text{中途退出的用户数}}{\text{问卷发放总数}} \times 100\%
\]

**符号表示**：

\[
R_{abandonment} = \frac{N_{abandoned}}{N_{distributed}} \times 100\%
\]

其中：
- \( N_{abandoned} \)：status='abandoned' 的记录数

**计算方法原理**：

1. **退出判定**：当用户关闭问卷页面或离开小程序时，如果 `isSubmitted=false` 且 `currentQuestionIndex >= 0`，则判定为中途退出
2. **记录退出位置**：`exit_question_index` 记录用户退出时的题目索引（0-based）
3. **统计放弃数**：查询 `user_sessions` 集合中 `status='abandoned'` 的记录数

**数据库查询方法**：

```javascript
// 实验组放弃数
const experimentalAbandoned = await db.collection('user_sessions')
  .where({
    group: 'experimental',
    status: 'abandoned'
  })
  .count();

// 放弃率计算
const experimentalAbandonmentRate = experimentalAbandoned / totalDistributed * 100;
```

**退出位置分布分析**：

```javascript
// 实验组退出位置分布
const experimentalExitDistribution = await db.collection('user_sessions')
  .where({
    group: 'experimental',
    status: 'abandoned'
  })
  .field({
    exit_question_index: true,
    has_skip: true
  })
  .get();

// 分析退出集中位置
const exitIndexCounts = {};
experimentalExitDistribution.data.forEach(record => {
  const index = record.exit_question_index;
  exitIndexCounts[index] = (exitIndexCounts[index] || 0) + 1;
});
```

**统计意义**：
- 放弃率反映问卷的难度和用户坚持度
- 实验组预期放弃率低于对照组，假设LLM辅助能降低填写门槛

---

### 3.3 跳过率（Skip Rate）

**定义**：存在漏填必填题的用户占发放总数的比例。

**计算公式**：

\[
\text{跳过率} = \frac{\text{有漏填行为的用户数}}{\text{问卷发放总数}} \times 100\%
\]

**符号表示**：

\[
R_{skip} = \frac{N_{skipped}}{N_{distributed}} \times 100\%
\]

其中：
- \( N_{skipped} \)：`has_skip=true` 的记录数

**计算方法原理**：

1. **跳过判定**：
   - 对于已提交问卷（`status='completed'`）：理论上 `has_skip` 应该为 `false`（程序验证机制）
   - 对于中途退出（`status='abandoned'`）：检查已填写题目是否包含所有必填题
2. **必填题定义**：问卷中 `required=true` 的题目（共30题，第31题为非必填）
3. **跳过标记**：程序在提交或退出时检查必填题填写情况，设置 `has_skip` 标志

**数据库查询方法**：

```javascript
// 实验组跳过数（包括中途退出和已提交但有跳过）
const experimentalSkipped = await db.collection('user_sessions')
  .where({
    group: 'experimental',
    has_skip: true
  })
  .count();

// 跳过率计算
const experimentalSkipRate = experimentalSkipped / totalDistributed * 100;
```

**统计意义**：
- 跳过率反映问卷的填写规范性和用户配合度
- 由于程序强制验证，已完成问卷的跳过率预期为0
- 主要分析中途退出用户的跳过行为

---

## 四、准确性指标

### 4.1 矛盾率（Contradiction Rate）

**定义**：填写答案存在逻辑矛盾的问卷占发放总数的比例。

**计算公式**：

\[
\text{矛盾率} = \frac{\text{存在逻辑矛盾的问卷数}}{\text{问卷发放总数}} \times 100\%
\]

**符号表示**：

\[
R_{contradiction} = \frac{N_{contradiction}}{N_{distributed}} \times 100\%
\]

其中：
- \( N_{contradiction} \)：人工判定存在逻辑矛盾的问卷数

**计算方法原理**：

由于逻辑矛盾需要人工判断，本指标采用**半自动 + 人工审核**的方式：

1. **数据提取**：从 `fill_record` 集合提取所有已提交问卷的答案
2. **规则定义**：基于问卷题目间的逻辑关系定义矛盾规则
3. **人工审核**：医护人员逐份检查答案的逻辑一致性

**矛盾规则示例**：

| 规则编号 | 涉及题目 | 矛盾描述 |
|----------|----------|----------|
| R1 | Q6（家族史） vs Q7（冠心病早发） | Q6选"以上皆无"但Q7选"是" |
| R2 | Q6（家族史） vs Q8（自身患病） | Q6选某疾病但Q8选"以上皆无"（同一疾病） |
| R3 | Q7（冠心病早发） vs Q9（长期服药） | Q7选"是"但Q9未选"降脂药" |
| R4 | Q8（自身患病） vs Q9（长期服药） | Q8选"高血压"但Q9未选"降压药" |
| R5 | Q4（性别） vs Q32/Q33（女性题） | 男性填写了Q32或Q33 |
| R6 | Q12（吸烟） vs Q13（饮酒） | 与流行病学常识明显矛盾的极端情况 |

**数据库查询方法**：

```javascript
// 提取所有已提交问卷答案
const filledRecords = await db.collection('fill_record')
  .where({
    status: 'completed'
  })
  .get();

// 导出为CSV格式供人工审核
const csvData = filledRecords.data.map(record => ({
  record_id: record.record_id,
  group: record.group,
  ...record.answers
}));
```

**人工审核流程**：

1. **问卷编号**：为每份问卷分配唯一审核编号
2. **规则对照**：审核员按上述规则逐一检查
3. **矛盾标记**：发现矛盾记录规则编号
4. **交叉验证**：随机抽取10%由第二名审核员复核

**统计意义**：
- 矛盾率反映填写数据的可靠性和逻辑一致性
- 实验组预期矛盾率低于对照组，假设LLM逻辑检查功能能有效减少矛盾

---

### 4.2 修正率（没有）（Corrected Rate，仅实验组）

**定义**：LLM逻辑检查中，用户采纳建议并修正答案的比例。

**计算公式**：

\[
\text{修正率} = \frac{\text{用户采纳建议的次数}}{\text{LLM逻辑检查总次数}} \times 100\%
\]

**符号表示**：

\[
R_{corrected} = \frac{N_{accept}}{N_{logic\_check}} \times 100\%
\]

其中：
- \( N_{accept} \)：`event_type='logic_check'` 且 `user_action='accept'` 的记录数
- \( N_{logic\_check} \)：`event_type='logic_check'` 的记录总数

**计算方法原理**：

1. **LLM逻辑检查触发**：当用户完成题目填写时，系统后台调用LLM检查答案逻辑一致性
2. **用户决策**：系统弹窗显示矛盾提示，用户可选择"采纳建议"（accept）、"忽略提示"（reject）或"暂不处理"（ignore）
3. **行为记录**：`llm_interactions` 集合记录用户的决策

**数据库查询方法**：

```javascript
// LLM逻辑检查总次数
const logicCheckCount = await db.collection('llm_interactions')
  .where({
    event_type: 'logic_check'
  })
  .count();

// 用户采纳建议次数
const acceptCount = await db.collection('llm_interactions')
  .where({
    event_type: 'logic_check',
    user_action: 'accept'
  })
  .count();

// 修正率计算
const correctedRate = acceptCount / logicCheckCount * 100;
```

**行为分布分析**：

```javascript
// 用户行为分布
const behaviorDistribution = await db.collection('llm_interactions')
  .where({
    event_type: 'logic_check'
  })
  .group('user_action')
  .get();

// 预期输出
// { user_action: 'accept', count: 45 }
// { user_action: 'reject', count: 30 }
// { user_action: 'ignore', count: 25 }
```

**统计意义**：
- 修正率反映LLM逻辑检查的实用性和用户接受度
- 高修正率说明LLM检测到的矛盾是有价值的，用户愿意采纳
- 低修正率可能说明：LLM误报率高，或用户对AI建议不信任

## 五、效率性指标

### 5.1 平均填写时长（Average Duration）

**定义**：完成问卷的平均耗时。

**计算公式**：

\[
\text{平均时长} = \frac{\sum \text{所有完成问卷的填写时长}}{\text{完成问卷数}}
\]

**符号表示**：

\[
\overline{T} = \frac{\sum_{i=1}^{N_{completed}} T_i}{N_{completed}}
\]

其中：
- \( T_i \)：第i份问卷的填写时长（毫秒）
- \( N_{completed} \)：完成问卷数

**计算方法原理**：

1. **计时机制**：问卷进入页面时记录 `start_time`，提交时记录 `end_time`
2. **时长计算**：`duration = end_time - start_time`（单位：毫秒）
3. **统计平均值**：对所有已完成问卷的 `duration` 求平均

**数据库查询方法**：

```javascript
// 实验组所有完成问卷
const experimentalSessions = await db.collection('user_sessions')
  .where({
    group: 'experimental',
    status: 'completed'
  })
  .get();

// 计算平均时长（毫秒 → 秒）
const totalDuration = experimentalSessions.data.reduce((sum, session) => {
  return sum + session.duration;
}, 0);

const avgDurationMs = totalDuration / experimentalSessions.data.length;
const avgDurationSec = avgDurationMs / 1000; // 转换为秒
```

**标准化转换**：

```javascript
// 转换为分钟
const avgDurationMin = avgDurationSec / 60;

// 格式化输出（分钟:秒）
const minutes = Math.floor(avgDurationSec / 60);
const seconds = Math.round(avgDurationSec % 60);
console.log(`平均填写时长：${minutes}分${seconds}秒`);
```

**统计意义**：
- 平均时长反映填写效率
- 实验组预期时长可能低于或高于对照组（需分析LLM交互是否节省时间）
- 如果LLM能有效解释问题，可能减少理解时间，但交互本身会增加时间

---

### 5.2 中位数填写时长（Median Duration）

**定义**：填写时长的中位数，反映典型填写时间。

**计算公式**：

\[
\text{中位数} =
\begin{cases}
T_{(k+1)} & \text{若} N \text{为奇数} \\
\frac{T_{(k)} + T_{(k+1)}}{2} & \text{若} N \text{为偶数}
\end{cases}
\]

其中：
- \( T_{(i)} \)：排序后的第i个时长
- \( k = \lfloor N/2 \rfloor \)

**计算方法原理**：

1. **提取所有时长**：从 `user_sessions` 提取所有 `status='completed'` 的 `duration`
2. **排序**：按时长从小到大排序
3. **取中位数**：根据样本量取中间值

**数据库查询方法**：

```javascript
// 提取所有时长
const experimentalSessions = await db.collection('user_sessions')
  .where({
    group: 'experimental',
    status: 'completed'
  })
  .field({
    duration: true
  })
  .get();

// 排序
const durations = experimentalSessions.data
  .map(session => session.duration)
  .sort((a, b) => a - b);

// 计算中位数
const n = durations.length;
let medianDurationMs;

if (n % 2 === 1) {
  // 奇数个样本，取中间值
  medianDurationMs = durations[Math.floor(n / 2)];
} else {
  // 偶数个样本，取中间两个的平均
  medianDurationMs = (durations[n / 2 - 1] + durations[n / 2]) / 2;
}

// 转换为秒
const medianDurationSec = medianDurationMs / 1000;
```

**统计意义**：
- 中位数不受极端值影响，比平均值更能反映典型填写时间
- 如果存在超长或超短的异常填写，中位数比平均值更稳健
- 对比平均值和中位数的差异可以判断数据分布是否对称

## 六、交互性指标（仅实验组）

- \( N_{logic\_check} \)：逻辑检查次数

**统计意义**：
- 反映LLM功能的实际使用频率
- 高频使用说明用户依赖LLM辅助

---

### 6.2 逻辑检查次数

**定义**：系统自动触发的LLM逻辑检查次数。

**计算公式**：

\[
N_{logic\_check} = \text{count}(event\_type = 'logic\_check')

**数据库查询方法**：

const logicCheckCount = await db.collection('llm_interactions')

**统计意义**：
- 反映LLM逻辑检查的触发频率

### 6.3 AI问答次数

**定义**：用户主动发起的AI问答次数。

**计算公式**：

\[

- 反映用户对AI解释功能的依赖程度

### 6.4 交互类型分布

**定义**：不同交互类型（逻辑检查 vs AI问答）的占比。

**计算公式**：

\[
P_{logic\_check} = \frac{N_{logic\_check}}{N_{llm\_calls}} \times 100\%
\]

\[
P_{ai\_question} = \frac{N_{ai\_question}}{N_{llm\_calls}} \times 100\%
\]

**数据库查询方法**：

```javascript

  .group('event_type')
  .get();

// 预期输出

// { event_type: 'ai_question', count: 25 }
```

**统计意义**：
- 分析用户主要使用哪种LLM功能
- 逻辑检查占比高说明矛盾检测功能更实用
- AI问答占比高说明问题解释需求更大

## 七、统计分析方法

### 7.1 描述性统计

| 指标类型 | 统计量 | 适用场景 |
|----------|--------|----------|
| 计数型（完成率等） | 频数、百分比 | 应答率指标 |
| 计量型（时长等） | 均值、标准差、中位数、四分位数 | 效率性指标 |

### 7.2 推断性统计

| 指标对比 | 统计方法 | 前提条件 |
|----------|----------|----------|
| 完成率对比 | 卡方检验（χ²检验） | 独立样本，分类变量 |
| 时长对比 | t检验（正态分布）或Mann-Whitney U检验（非正态） | 独立样本，连续变量 |
| 矛盾率对比 | 卡方检验（χ²检验） | 独立样本，分类变量 |

### 7.3 统计显著性检验

**假设检验示例（完成率）**：

\[
H_0: \text{实验组完成率} = \text{对照组完成率}
\]

\[
H_1: \text{实验组完成率} \neq \text{对照组完成率}
\]

**卡方检验**：

\[
\chi^2 = \sum \frac{(O_{ij} - E_{ij})^2}{E_{ij}}
\]

其中：
- \( O_{ij} \)：观测频数
- \( E_{ij} \)：期望频数

**结果解读**：
- P < 0.05：拒绝原假设，两组差异显著
- P ≥ 0.05：无法拒绝原假设，两组差异不显著

| 指标 | 预期实验组 | 预期对照组 | 假设方向 |
|------|-----------|-----------|----------|
| 矛盾率 | < 对照组 | - | LLM逻辑检查减少矛盾 |

|------|-----------|-----------|----------|

| 中位数时长 | ≈ 或略 > 对照组 | - | 交互时间 vs 解释时间 |

| 指标 | 预期范围 | 说明 |
|------|----------|------|
| LLM调用总次数 | 3-8次/用户 | 假设每用户交互次数 |
| 逻辑检查次数占比 | 60%-80% | 主要由系统触发 |
| AI问答次数占比 | 20%-40% | 用户主动发起 |
| 修正率 | 50%-70% | 合理接受度 |

## 十、局限性说明

### 10.1 数据局限性

1. **分母定义**：发放总数需人工统计，可能存在统计误差
2. **矛盾率依赖人工审核**：主观性较强，需要建立明确的审核标准
3. **时长影响因素**：用户填写习惯、网络延迟等外部因素可能影响时长

### 10.2 指标局限性

1. **完成率**：无法区分"主动完成"和"应付完成"
2. **矛盾率**：仅检测已定义的矛盾规则，可能遗漏复杂逻辑问题
3. **修正率**：无法区分"必要修正"和"非必要修正"

### 10.3 外部效度

1. 本研究对象为糖尿病患者，结果可能不适用于其他人群
2. 问卷长度为32题，可能不适用于不同长度的问卷
3. 微信小程序平台特性，可能影响用户行为模式
